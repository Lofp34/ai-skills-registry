# Guide de Rédaction des Frameworks

## Structure Type d'un Framework Performant

Ce document définit la structure standard à suivre pour rédiger chaque framework de la bibliothèque Elite Business Advisory. Cette structure a été extraite de l'analyse des frameworks existants (BANT, SPIN, SWOT, MECE, RACI, BATNA, Sales Funnel, Value Proposition Canvas, Impact-Effort Matrix).

---

## Structure en 12 Sections

### 1. En-tête (Titre + Tagline)

```markdown
# [NOM DU FRAMEWORK] Framework

**[Acronyme développé ou description courte]**

---
```

**Exemple :**
```markdown
# BANT Framework - Qualification de Prospects

**Budget, Authority, Need, Timeline**

---
```

---

### 2. Vue d'Ensemble (Métadonnées)

```markdown
## 📋 Vue d'Ensemble

**Origine :** [Auteur/Cabinet] ([Année])
**Usage :** [Cas d'usage principal en 1 phrase]
**Niveau :** [Essentiel / Avancé] - [Quand l'utiliser]
**Expert :** [Marie Durand (McKinsey) / Thomas Bernard (Bain) / Sophie Martineau (BCG)]
```

**Règles :**
- **Origine** : Toujours citer la source (auteur, cabinet, année)
- **Usage** : 1 phrase claire et actionnable
- **Niveau** : "Essentiel" pour les frameworks fondamentaux, "Avancé" pour les spécialisés
- **Expert** : Choisir l'expert le plus pertinent selon la catégorie :
  - Marie Durand (McKinsey) → Structuration, Stratégie, Analyse
  - Thomas Bernard (Bain) → Vente, Exécution, Pilotage
  - Sophie Martineau (BCG) → Juridique, Contractuel, Risques

---

### 3. Définition du Framework

```markdown
## 🎯 Qu'est-ce que [NOM] ?

[Description claire en 2-3 paragraphes]

### Les [X] Dimensions/Étapes/Critères

[Tableau ou schéma ASCII présentant les composantes]
```

**Règles :**
- Commencer par une définition simple (1-2 phrases)
- Ajouter le **principe clé** ou la **règle fondamentale**
- Inclure un **schéma visuel** (ASCII ou tableau) pour visualiser le framework

**Exemple de schéma ASCII :**
```
      IMPACT
        │
   HIGH │   PROJETS    │   QUICK WINS
        │   MAJEURS    │   (Priorité 1)
────────┼──────────────┼────────────────► EFFORT
        │   TIME       │   FILL-INS
   LOW  │   WASTERS    │   (Remplissage)
```

---

### 4. Détail des Composantes

```markdown
## 🔍 Les [X] [Composantes] en Détail

### 1️⃣ [Première Composante]

**Définition/Question clé :** [...]

**Ce qu'on cherche :**
- Point 1
- Point 2
- Point 3

**Exemples pour un consultant :**
- ✅ Exemple positif 1
- ✅ Exemple positif 2
- ⚠️ Exemple de vigilance

**Questions à se poser :**
- Question 1 ?
- Question 2 ?

---

### 2️⃣ [Deuxième Composante]
[Même structure...]
```

**Règles :**
- Utiliser des émojis numérotés (1️⃣, 2️⃣, 3️⃣...)
- Structure identique pour chaque composante
- Inclure des **exemples concrets** liés au conseil/vente
- Ajouter des **signaux positifs ✅** et **négatifs ❌**

---

### 5. Exemple Concret Complet

```markdown
## 📊 Exemple Concret : [Situation Réaliste]

### Contexte
[Description de la situation en 2-3 lignes]

### Application du Framework

[Tableau ou étapes détaillées montrant l'application]

### Résultat
[Ce que le framework permet de conclure/décider]
```

**Règles :**
- Utiliser un contexte proche du consultant/formateur
- Montrer l'application pas-à-pas
- Conclure par une **décision claire**

---

### 6. Guide d'Utilisation Pratique

```markdown
## 📈 Comment Utiliser [NOM] ?

### Étape 1 : [Titre de l'étape]

**Méthode :**
[Description de l'étape]

**Astuce :** [Conseil pratique]

---

### Étape 2 : [Titre]
[...]

### Étape 3 : [Titre]
[...]
```

**Règles :**
- 3 à 5 étapes maximum
- Chaque étape = 1 action concrète
- Ajouter des **astuces** et **bonnes pratiques**

---

### 7. Variantes et Options Avancées

```markdown
## 💡 Variantes et Astuces

### Variante 1 : [Nom de la variante]

[Description et quand l'utiliser]

### Variante 2 : [Nom]
[...]
```

**Règles :**
- Optionnel si le framework a des variantes connues
- Mentionner les extensions (ex: RACI → RACI-VS, RASCI)

---

### 8. Erreurs Fréquentes

```markdown
## ⚠️ Erreurs Fréquentes / Pièges à Éviter

### ❌ Erreur 1 : [Titre de l'erreur]

**Symptôme :** [Comment on la détecte]

**Problème :** [Conséquence négative]

**Solution :** [Comment l'éviter]

---

### ❌ Erreur 2 : [Titre]
[Même structure...]
```

**Règles :**
- 3 à 5 erreurs maximum
- Structure identique : Symptôme → Problème → Solution
- Erreurs issues de l'expérience réelle (pas théoriques)

---

### 9. Checklist de Validation

```markdown
## ✅ Checklist [NOM]

**[Catégorie 1] :**
- [ ] Point de vérification 1
- [ ] Point de vérification 2
- [ ] Point de vérification 3

**[Catégorie 2] :**
- [ ] Point 1
- [ ] Point 2
```

**Règles :**
- Format checklist avec cases à cocher [ ]
- Regrouper par catégorie (Préparation, Exécution, Suivi)
- 8 à 15 points maximum

---

### 10. Citation Expert

```markdown
## 💡 Conseil [Prénom] ([Cabinet])

> *"[Citation inspirante et pratique de l'expert, 2-4 phrases. Doit être actionnable et mémorable.]"*
```

**Règles :**
- Toujours en italique entre guillemets
- Ton direct et pragmatique
- Mentionner une règle simple ou un conseil clé

---

### 11. Frameworks Complémentaires

```markdown
## 🔗 Frameworks Complémentaires

**Utiliser [NOM] avec :**
- **[Framework 1]** : [Pourquoi les combiner]
- **[Framework 2]** : [Pourquoi les combiner]
- **[Framework 3]** : [Pourquoi les combiner]
```

**Règles :**
- 2 à 4 frameworks liés
- Expliquer brièvement le lien

---

### 12. Pour Aller Plus Loin (optionnel)

```markdown
## 📚 Pour Aller Plus Loin

**Lectures recommandées :**
- "[Titre du livre]" - [Auteur]
- [Articles, ressources]

**Outils :**
- [Outils pratiques pour appliquer le framework]

---

**Prochains frameworks à étudier :**
- [Framework suggéré 1]
- [Framework suggéré 2]
```

## Conventions de Style

### Émojis Standards

| Usage | Émoji |
|-------|-------|
| Vue d'ensemble | 📋 |
| Définition/Objectif | 🎯 |
| Détail/Analyse | 🔍 |
| Exemple/Données | 📊 |
| Guide pratique | 📈 |
| Conseil/Astuce | 💡 |
| Erreur/Piège | ⚠️ |
| Checklist | ✅ |
| Liens | 🔗 |
| Ressources | 📚 |
| Positif | ✅ 🟢 |
| Négatif | ❌ 🔴 |
| Attention | ⚠️ |

### Formatage

- **Gras** pour les termes clés et les titres dans le texte
- *Italique* pour les citations et les emphases
- `Code` pour les formules et les noms techniques
- Tableaux pour les comparaisons et les matrices
- Listes à puces pour les énumérations
- Blocs de code pour les schémas ASCII

### Ton et Style

- **Ton** : Professionnel mais accessible, comme un consultant senior qui explique à un junior
- **Voix** : Active, directe ("Faites X" plutôt que "Il faudrait faire X")
- **Exemples** : Toujours liés au contexte consultant/formateur/vente B2B
- **Longueur** : 300-400 lignes par framework (ni trop court, ni exhaustif)

## Checklist Avant Publication

Avant de publier un nouveau framework, vérifier :

- [ ] Les 12 sections sont présentes (ou justifiées si absentes)
- [ ] L'expert assigné correspond à la catégorie du framework
- [ ] Au moins 1 exemple concret avec contexte consultant/formateur
- [ ] 3-5 erreurs fréquentes documentées
- [ ] Checklist de validation incluse
- [ ] Citation expert mémorable et actionnable
- [ ] Liens vers 2-4 frameworks complémentaires
- [ ] Longueur entre 300-400 lignes
- [ ] Émojis cohérents avec les conventions
- [ ] Pas de fautes d'orthographe
